## Migration 패턴 학습 — 핵심 통찰

### ngrok은 진짜 이전이 아니다

PC → ngrok → Render → GitHub Actions 이전 과정에서:

- ngrok 단계는 단순 url 터널일 뿐 코드/DB/환경변수가 그대로 본인 PC에 있는 상태
- "ngrok에서 잘 작동"이 다음 단계 검증 가치를 갖지 못함
- 본인 표현: "ngrok도 작동 확률 많았겠지 단지 url 오픈해준거니까" — 정확한 진단

### 진짜 이전은 Render와 Actions에서 발생

- Render: 첫 외부 머신, gunicorn으로 Flask 운영. 코드·의존성·환경변수 모두 새로 설정 필요
- Actions: 매 실행마다 빈 환경. DB 스키마부터 새로 보장해야 함

### Migration 시 빠뜨리기 쉬운 3가지 (체크리스트)

1. **DB 스키마** — 매 환경마다 ALTER 보장 (PC는 한 번 만들면 끝, 외부 머신은 매번)
2. **환경변수** — 각 외부 시스템에 별도 설정 (Render Dashboard ≠ GitHub Secrets)
3. **가시화** — 실패가 진짜 실패로 표시되는지 확인, 거짓 success 차단
