## 핵심 교훈

1. **백필 검증의 함정** — 5/8 백로그 035 dry-run 에서 4 sample (4/23, 4/24, 5/8, bad spSeq) 모두 통과했지만, **모두 같은 detail 페이지의 같은 위치의 selector 결과** 였음. 다양한 spSeq 의 cross-match 검증이 빠져있어 selector 오작동을 못 잡음. 진단 신호("동일 title 두 row" → 5/8 일지 line 79-80)가 있었으나 분석 안 했음.

2. **사용자 시각의 중요성** — 사용자가 운영 사이트와 원본을 비교해서 "거의 일치하지 않음" 한 마디 → 5/8 chain (위젯 #1 깨진 제목 → DB stale → proxy down) 패턴 또 한 번 반복.

3. **알림 정책과 status sync 의 분리** — `filter_open_announcements` 가 메일/알림 흐름을 위한 필터이지만, status sync 도 같은 path 를 쓰면서 마감 사업의 status 갱신이 막혔음. 두 흐름 분리가 정답 (sync_status_to_db 별도 함수).

4. **selector 부수성 (parent 추적)** — `span.titleView` 같은 모호한 class 만으로 selector 잡으면 본문/nav 구분이 불가. 항상 parent chain (`ul.board_bottom` 등) 을 확인해서 의도한 위치인지 검증해야 함.

## 핵심 교훈 (049 + 050)

1. **백필 단계의 함정 (반복)** — 백로그 034 가 컬럼 ALTER + crawler 출력은 했지만
   update_db 의 INSERT/UPDATE SQL 컬럼 목록 갱신을 빠트려 사실상 dead column.
   "code applied" ≠ "data flowing". 다음 컬럼 추가 작업부터는 update_db._prepare_row
   + _upsert_one 의 INSERT/UPDATE SQL 동시 점검 체크리스트 필요.

2. **임시 필터의 수명 관리** — 위젯 organization='전북수출통합지원시스템' 필터는
   2026-05-07 jbbi 섞임 회피용 임시 워크어라운드였지만, 백로그 041 (jbbi misclassify
   해결) 이후 제거 안 되고 살아남아 049 fix 의 1건 누락 사고 유발. 임시 필터 마다
   "제거 조건 (백로그 041 완료 시)" 댓글 달면 추적 쉬움.

3. **머지 로직 일반 패턴** — 049 (notice_chk/notice_order) 와 050 (organization)
   둘 다 "새 값이 누락/fallback 인데 옛 값이 진짜면 보존" 패턴. 향후 신규 컬럼 추가
   시 _upsert_one 머지 로직 기본 점검 항목으로 추가.

4. **Phase 6 stop 패턴 효과** — 자동 진행이 아니라 옵션 비교표 + 시뮬레이션 + 사용자
   결정 stop 으로 나눈 게 좋았음. 옵션 C 가 실은 위젯 깨짐인 걸 사용자가 결정 단계
